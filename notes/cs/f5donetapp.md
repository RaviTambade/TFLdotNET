# What Really Happens After Pressing F5 in a .NET Application?

Most students think:

> "I wrote C# code, pressed F5, and my application started."

As a Solution Developer, that understanding is not enough.

When your application behaves unexpectedly, consumes memory, becomes slow, or crashes in production, you must understand what happens behind the scenes.

Let's travel inside the .NET runtime and follow the journey of your code.


# Stage 1: Writing Source Code

You create files such as:

```csharp
class Program
{
    static void Main()
    {
        Console.WriteLine("Hello World");
    }
}
```

At this stage, it is only human-readable text.

The CPU cannot execute C#.

Something must translate it.


# Stage 2: Roslyn Compiler Converts C# to IL

When you build the project:

```bash
dotnet build
```

The .NET Compiler Platform (Roslyn) starts working.

### Input

```csharp
Console.WriteLine("Hello World");
```

### Output

```text
IL (Intermediate Language)
```

Think of IL as:

> A universal language understood by every .NET runtime.


## Why IL?

Suppose you create:

```text
MyApplication.dll
```

The same DLL can run on:

* Windows
* Linux
* macOS

because IL is platform independent.

This is one of the biggest strengths of .NET.

### Student Analogy

Think of IL as English.

Different countries speak different native languages.

But everyone can understand English.

Similarly:

```text
C# → IL → Native Machine Code
```

# Stage 3: Assembly Creation

After compilation, .NET generates:

```text
MyApplication.dll
```

or

```text
MyApplication.exe
```

Inside the assembly:

### IL Code

Instructions generated by compiler.

### Metadata

Information about:

```csharp
Classes
Methods
Properties
Interfaces
Attributes
Generics
```

Example:

```csharp
class Customer
{
    public string Name { get; set; }
}
```

Metadata stores:

```text
Type: Customer
Property: Name
Type: String
```

This metadata enables:

* Reflection
* Dependency Injection
* Entity Framework
* ASP.NET Routing
* Serialization

Without metadata, modern .NET would not exist.


# Stage 4: CoreCLR Starts

Now you press:

```text
F5
```

Visual Studio launches the .NET Runtime.

The heart of runtime is:

**CoreCLR**

CoreCLR behaves like the Operating Manager of your application.


## CoreCLR Responsibilities

### 1. Load Assemblies

```text
Program.dll
BusinessLogic.dll
DataAccess.dll
Newtonsoft.Json.dll
```

All dependencies are loaded.


### 2. Load Metadata

Runtime discovers:

```text
Classes
Methods
Interfaces
Attributes
```

Everything is registered internally.



### 3. Initialize Garbage Collector

GC starts monitoring memory.

It prepares:

```text
Gen0
Gen1
Gen2
```

memory regions.


### 4. Initialize Thread Pool

The runtime creates infrastructure for:

```csharp
async
await
Task
Parallel
```

operations.


# Stage 5: JIT Compilation Begins

At this point IL still cannot run on CPU.

CPU understands only machine code.

Now enters:

**JIT (Just-In-Time Compiler)**


## Important Question

Does JIT compile the entire application?

No.

This is where many developers are mistaken.

JIT compiles only when a method is called for the first time.


Example:

```csharp
Main()
{
    CalculateSalary();
}
```

When execution reaches:

```csharp
CalculateSalary();
```

JIT converts that method from IL into native code.

## Visualization

```text
C# Source
     ↓
Roslyn
     ↓
IL Code
     ↓
CoreCLR
     ↓
JIT
     ↓
Native Machine Code
     ↓
CPU
```

# Why First Request Is Slower?

Imagine an ASP.NET API.

First request:

```text
Load Assembly
Load Metadata
JIT Compile
Execute
```

Second request:

```text
Execute
```

No JIT required.

Therefore:

```text
First Request = Slow
Later Requests = Fast
```

# Modern Optimization: Tiered Compilation

.NET uses a smart strategy.

## Tier 0

Compile quickly.

```text
Fast startup
Less optimization
```

## Tier 1

After runtime observes frequent execution:

```text
Method is Hot
```

it recompiles.

```text
More optimization
Higher performance
```

## Dynamic PGO

Profile Guided Optimization.

Runtime observes:

```text
Which code executes most?
Which branches are common?
```

and optimizes accordingly.

Think of it as:

> The runtime learning application behavior while it runs.


# Native AOT

Traditional .NET:

```text
IL + JIT
```

Native AOT:

```text
C# → Native Machine Code
```

during build.

No JIT.

Benefits:

* Faster startup
* Lower memory
* Smaller runtime footprint

Tradeoff:

* Less dynamic behavior
* Reflection limitations
* No runtime code generation

# Stage 6: Actual Execution

Now machine code is executing on CPU.

But many invisible workers continue operating.

# Worker 1: Garbage Collector (GC)

Developers create objects:

```csharp
var customer = new Customer();
```

GC tracks them.

## Generation 0

Short-lived objects.

Example:

```csharp
string temp = "Hello";
```

Most objects die here.

```text
Fast Collection
```

## Generation 1

Temporary holding area.

Objects surviving Gen0 move here.


## Generation 2

Long-lived objects.

Examples:

```csharp
Cache
Singleton
Application Services
```

Collections here are expensive.

## Why Latency Spikes Occur

Suppose application pauses unexpectedly.

Often reason is:

```text
Gen2 Collection
```

because GC must inspect a huge amount of memory.

When performance engineers investigate:

```text
GC Time %
Gen2 Count
Allocation Rate
```

these are among the first metrics they check.

# Worker 2: Thread Pool

Every ASP.NET application heavily uses the Thread Pool.

Example:

```csharp
await repository.GetCustomersAsync();
```

Students often think:

```text
await = new thread
```

Wrong.


Actual Flow:

```text
Thread Executes
      ↓
Hits await
      ↓
Returns Thread to Pool
      ↓
I/O Continues
      ↓
Operation Completes
      ↓
Pool Selects Available Thread
      ↓
Execution Resumes
```

This is why async scales so well.

Thousands of requests can be managed using relatively few threads.

# Complete Journey of a .NET Application

```text
Write C# Code
        ↓
Roslyn Compiler
        ↓
Intermediate Language (IL)
        ↓
Assembly (.dll/.exe)
        ↓
CoreCLR Loads Assembly
        ↓
Metadata Loading
        ↓
GC Initialization
        ↓
Thread Pool Initialization
        ↓
JIT Compilation
        ↓
Native Machine Code
        ↓
CPU Execution
        ↓
GC + Thread Pool Monitor Continuously
```

# Mentor's Industry Insight

A beginner asks:

> "My application is slow. Why?"

An experienced developer asks:

```text
Is startup slow?
→ Check Assembly Loading

Is first request slow?
→ Check JIT

Are there random pauses?
→ Check GC

High memory usage?
→ Check Object Lifetime

Poor scalability?
→ Check Thread Pool and Async Usage

Slow API response?
→ Check Database, Network, and GC Metrics
```

This is the transition from **Programmer** to **Solution Developer**.

The day you understand what happens after pressing **F5**, debugging stops being guesswork and starts becoming engineering.
