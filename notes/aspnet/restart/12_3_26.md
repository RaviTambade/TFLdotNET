# Preparing ASP.NET Framework Developers for ASP.NET Core in the AI Era

Mentor is **mentoring ASP.NET Framework developers and preparing them for the ASP.NET Core world**, while also discussing **AI tools, GitHub Copilot, and modern development practices**. 

Hello everyone 👋

Today I want to talk about something very important for every **ASP.NET developer**.

Many of you already know **ASP.NET Framework, WebForms, or ASP.NET MVC**.
But the industry is moving toward **ASP.NET Core, Cloud, and AI-assisted development**.

So the question is not:

> “Should we learn ASP.NET Core?”

The real question is:

> “How do we evolve as developers in the era of AI and modern frameworks?”

## 1. Software Development is Changing ⚡

Earlier a developer used to write **almost 100% code manually**.

Today the situation is changing.

With tools like:

* Visual Studio
* GitHub Copilot
* GitHub

AI can generate **60–70% of the boilerplate code**.

That means:

* Controllers
* CRUD APIs
* Models
* Service classes
* Basic validations

can be **generated automatically**.

But the remaining **30–40% is the real engineering work**.

That includes:

* Domain understanding
* Architecture decisions
* Performance
* Debugging production issues
* Security
* Integration with other systems

This is where **real developers are needed**.

## 2. Frameworks Already Reduced Boilerplate

Even before AI, frameworks were already helping us.

For example in **ASP.NET MVC**:

Instead of writing everything from scratch, frameworks introduced:

* MVC pattern
* Routing
* Model binding
* Dependency Injection
* Filters
* Middleware

These frameworks embed **industry best practices**.

Example:

Instead of manually creating objects like this:

```
ProductService service = new ProductService();
```

Modern frameworks use **Dependency Injection**:

```
public ProductController(IProductService service)
{
    _service = service;
}
```

The **IOC container automatically injects dependencies**.

This makes applications:

* loosely coupled
* testable
* maintainable

Key benefits:

- ✔ Cloud-ready

- ✔ Lightweight architecture
- ✔ API-first development

It works perfectly with cloud platforms like:

* Microsoft cloud services
* Microsoft Azure

# Mentor Session: Understanding MVC Controller vs Web API Controller

Mentor is  explaining to a student the difference between **ASP.NET MVC controller and Web API controller**, and also showing how to **create a project using .NET CLI and test APIs using Postman**.

Today we are building a **CRM Web Application** and discussing how ASP.NET applications are structured.

We also created a sample project using **.NET CLI** and opened it in **VS Code**.

Tools used:

* Visual Studio Code
* Postman
* ASP.NET Core


# 1. Creating ASP.NET MVC Application using CLI

Instead of using GUI, we can create projects using **.NET CLI**.

Example command:

```bash
dotnet new mvc -o CRMWebApp
```

Explanation:

| Part         | Meaning              |
| ------------ | -------------------- |
| dotnet       | .NET CLI command     |
| new          | create new project   |
| mvc          | MVC project template |
| -o CRMWebApp | output folder        |

This will automatically generate a project structure.



# 2. ASP.NET MVC Project Structure

After running the command, the project will contain folders like:

```
CRMWebApp
 ├── Controllers
 ├── Models
 ├── Views
 ├── wwwroot
 ├── Program.cs
 └── appsettings.json
```

### MVC pattern

| Folder      | Responsibility   |
| ----------- | ---------------- |
| Models      | Business data    |
| Views       | UI (HTML, Razor) |
| Controllers | Request handling |



# 3. MVC Controller Example

Example controller:

```csharp
public class HomeController : Controller
{
    public IActionResult Index()
    {
        return View();
    }
}
```

Important point:

This controller **inherits from**:

```csharp
Controller
```

This means it supports:

* View rendering
* HTML responses
* Razor pages

Typical flow:

```
Browser Request
      ↓
Controller Action
      ↓
View (.cshtml)
      ↓
HTML Response
```

So MVC is used for **web UI applications**.

# 4. API Controller Example

Now compare with Web API controller.

```csharp
[ApiController]
[Route("api/[controller]")]
public class CustomersController : ControllerBase
{
    [HttpGet]
    public IEnumerable<string> Get()
    {
        return new string[] { "Ravi", "Apurva" };
    }
}
```

Key differences:

| MVC Controller   | API Controller |
| ---------------- | -------------- |
| Controller       | ControllerBase |
| Returns Views    | Returns JSON   |
| UI rendering     | Data service   |
| Used in web apps | Used for APIs  |

# 5. Why API Controllers Exist

Modern applications often have **multiple frontends**:

* Web UI
* Mobile apps
* Desktop apps
* JavaScript SPA
* External systems

Instead of creating UI inside server, we expose **APIs**.

Example flow:

```
Mobile App
     ↓
HTTP Request
     ↓
Web API
     ↓
JSON Response
```

The response format is usually:

```json
{
  "id":1,
  "name":"Customer A"
}
```

This is called **Programmable UI**.

Instead of HTML, we return **data (JSON)**.

# 6. Service Oriented Applications

When we build APIs, we move toward **Service Oriented Architecture (SOA)**.

Architecture becomes:

```
Frontend Applications
(Web / Mobile / Desktop)

        ↓

ASP.NET Web API

        ↓

Business Logic

        ↓

Database
```

The frontend consumes APIs.

# 7. Testing API using Postman

We used:

Postman

Steps:

1. Start the application
2. API runs on port like

```
https://localhost:67890
```

3. Call API

```
GET /api/customers
```

4. Response:

```json
["Ravi","Apurva"]
```

# 8. Key Learning from the Session

Students must understand:

### MVC Application

* Controller returns **View**
* Used for **Web UI**

### API Application

* Controller returns **JSON**
* Used for **Services**

# 9. Modern Industry Approach

Today most systems follow:

```
Frontend (React / Angular / Mobile)

        ↓

ASP.NET Core Web API

        ↓

Business Logic

        ↓

Database
```

UI is handled by **JavaScript frameworks** while backend provides APIs.


# Final Mentor Message

Always remember:

MVC controller → **HTML UI**
API controller → **JSON data**

Both are important but used for **different purposes**.


# Mentor Session: Creating MVC vs Web API using .NET CLI

Today we created two types of applications using **.NET CLI** and opened them in:

* Visual Studio Code
* ASP.NET Core

The goal was to understand the **difference between MVC Web UI and REST API**.



# 1. Create MVC Application

Command:

```bash
dotnet new mvc -o TestMvcApp
```

Project structure generated automatically:

```
TestMvcApp
 ├── Controllers
 ├── Models
 ├── Views
 ├── wwwroot
 ├── Program.cs
 └── appsettings.json
```

Important folders:

| Folder      | Purpose                     |
| ----------- | --------------------------- |
| Controllers | Request handling            |
| Models      | Business data               |
| Views       | UI (.cshtml HTML templates) |

Example MVC controller:

```csharp
public class HomeController : Controller
{
    public IActionResult Index()
    {
        return View();
    }
}
```

When browser sends request:

```
http://localhost:5000/home/index
```

Flow:

```
Browser Request
     ↓
Controller
     ↓
View (.cshtml)
     ↓
HTML Page
```

So MVC applications generate **HTML UI**.


# 2. Create Web API Application

Command:

```bash
dotnet new webapi -o TestApi
```

Structure:

```
TestApi
 ├── Program.cs
 ├── appsettings.json
 └── WeatherForecast.cs
```

Minimal API example inside `Program.cs`:

```csharp
var app = builder.Build();

app.MapGet("/weatherforecast", () =>
{
    return new[] { "Sunny", "Rainy", "Cloudy" };
});

app.Run();
```

Run application:

```bash
dotnet run
```

Open browser:

```
http://localhost:5018/weatherforecast
```

Output:

```json
["Sunny","Rainy","Cloudy"]
```

This means the application returns **JSON data**, not HTML.


# 3. Web API with Controllers

If we want full controller structure:

```bash
dotnet new webapi --use-controllers -o TestRestApi
```

Now folder structure includes:

```
Controllers
   WeatherForecastController.cs
```

Example controller:

```csharp
[ApiController]
[Route("api/[controller]")]
public class CustomersController : ControllerBase
{
    [HttpGet]
    public IEnumerable<string> Get()
    {
        return new string[] { "Ravi", "Apurva" };
    }
}
```

API URL:

```
http://localhost:5018/api/customers
```

Response:

```json
["Ravi","Apurva"]
```

# 4. Running Project from CLI

Run project inside folder:

```bash
dotnet run
```

Run project from parent folder:

```bash
dotnet run --project ./TestApi
```

# 5. MVC vs Web API Concept

| Feature               | MVC Application | Web API        |
| --------------------- | --------------- | -------------- |
| Output                | HTML            | JSON           |
| UI                    | Server-side     | Client-side    |
| Used for              | Web UI          | Services       |
| Controller Base Class | Controller      | ControllerBase |


# 6. Key Idea You Explained in Class

You explained a **very important concept**:

If request returns **HTML UI → MVC Application**

If request returns **JSON Data → Web API**

Example:

```
MVC Response
-------------
<html>
   <body>Customer Page</body>
</html>
```

```
API Response
-------------
{
   "id":1,
   "name":"Customer"
}
```

This JSON response is called **Programmable UI**.

It can be used by:

* Web frontend
* Mobile apps
* Desktop applications
* JavaScript frameworks

# 7. Why Modern Systems Use APIs

Modern architecture:

```
Frontend (React / Angular / Mobile)

        ↓

ASP.NET Core Web API

        ↓

Business Logic

        ↓

Database
```

The backend provides **services**, not UI.
